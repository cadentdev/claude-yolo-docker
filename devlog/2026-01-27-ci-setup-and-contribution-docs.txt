❯ What do you think about requiring signed commits? Is that too burdensome  
  for an open source project?                                               

⏺ For most open source projects, signed commits are more burden than benefit.

  Why it's often skipped:
  - Contributors must set up GPG/SSH signing (extra friction)
  - Many casual contributors don't have signing configured
  - Deters first-time contributors who just want to fix a typo
  - PR review already verifies who submitted the change

  When it makes sense:
  - Security-critical projects (crypto libraries, package managers)
  - Projects where impersonation is a real threat
  - Compliance requirements

  For a Docker wrapper tool with PR-based workflow, the threat of commit
  spoofing is low. Anyone submitting a PR is already identified by their GitHub
  account, and you review before merging.

  I'd remove that task — it's adding friction without meaningful security gain
  for this project's threat model.

⏺ Practical options for this project:

  High value, easy to set up:
  - ShellCheck — Static analysis for claude-yo. Catches quoting bugs, undefined
  variables, portability issues. This alone catches most real problems.
  - bash -n claude-yo — Syntax validation (ShellCheck includes this)

  Medium value:
  - Hadolint — Dockerfile linter, checks best practices
  - Docker build test — Ensures the image actually builds (docker build -t test 
  .)
  - YAML schema validation — Validate .claude-yo.yml examples if you add any

  Low value for this project:
  - Markdown linting (docs aren't the product)
  - Security scanning of the image (it's just a base Python image + npm package)

  Simplest useful CI would be a GitHub Actions workflow that runs:
  - shellcheck claude-yo
  - docker build -t test .

  This catches syntax errors in the script and build failures in the Dockerfile.
   Takes ~1-2 minutes to run.
